# cfmerge

`cfmerge` - инструмент для разработчиков 1С, который помогает превратить файловую выгрузку основной конфигурации и файловую выгрузку одного расширения в одну обычную конфигурацию 1С.

Типовой вход:

```text
work/
  cf/   - файловая выгрузка основной конфигурации
  cfu/  - файловая выгрузка расширения
```

Результат работы:

```text
work/
  merged_cf/ - обычная файловая выгрузка конфигурации 1С
```

Идея простая: расширение хранит изменения отдельным слоем, а `cfmerge` материализует этот слой внутрь основной конфигурации. Получившийся каталог можно загрузить в тестовую базу через Конфигуратор, применить к базе и выгрузить в `.cf`.

## Что умеет инструмент

- Берет `cf/` как основу и не изменяет исходные каталоги.
- Добавляет собственные объекты расширения в основную конфигурацию.
- Не подменяет базовые XML-объекты заимствованными XML-обертками расширения.
- Переносит изменения модулей 1С:
  - `&Перед` и `&После` встраиваются в тело целевой процедуры;
  - `&Вместо` заменяет тело целевой процедуры или функции;
  - `&Вместо` с `ПродолжитьВызов()` сохраняет вызов исходного метода через внутренний помощник;
  - `&ИзменениеИКонтроль` применяется только при совпадении ожидаемого исходного текста.
- Поддерживает перенос обычных методов расширения, включая методы с директивами контекста и асинхронные процедуры.
- Выполняет слияние сложных управляемых форм:
  - добавление/изменение элементов, реквизитов, команд и условного оформления;
  - изменение порядка элементов;
  - перемещение элементов между группами;
  - изменение и удаление свойств элементов;
  - изменение обработчиков событий формы;
  - сохранение связности между событиями формы и BSL-обработчиками.
- Регенерирует `ConfigDumpInfo.xml`.
- Объединяет `ОтчетПоКонфигурации.txt`, если отчет есть и в `cf/`, и в `cfu/`. Если одного из отчетов нет, слияние отчета пропускается.
- Формирует JSON-отчет и человекочитаемый текстовый отчет.
- Выполняет базовые проверки XML и BSL.
- Может запускать локальные скрипты проверки через Конфигуратор 1С.
- Имеет графический интерфейс для выбора каталогов и запуска объединения без ручного ввода длинной команды.
- Поставляется с готовым Windows-файлом `dist/1C_cf_cfu_merge.exe` и скриптом для его пересборки.

## Что инструмент пока не умеет

- Не поддерживает несколько расширений одновременно и порядок их наложения.
- Не выполняет полное трехстороннее объединение для `&ИзменениеИКонтроль`, если исходный код основной конфигурации отличается от ожидаемого текста расширения. Перед выгрузкой конфигурации и расширения в файлы следует проверить возможность применения расширения стандартным механизмом платформы.
- Не является полноценным синтаксическим анализатором всего языка 1С. Анализ модулей ограничен тем, что нужно для переноса процедур, функций и аннотаций расширений.
- Не заменяет проверку Конфигуратором 1С. Успешные тесты Python и проверка XML не доказывают, что конфигурация применится к базе.


## Что понадобится

- Python 3.11 или новее.
- Платформа 1С:Предприятие 8.3 с `1cv8.exe`, если нужна финальная проверка загрузкой в базу (тестировалось на 8.3.27.1644).
- Тестовая файловая или серверная база 1С.
- Файловая выгрузка основной конфигурации.
- Файловая выгрузка одного расширения.

Проверки Python помогают быстро найти ошибки в структуре результата, но финальную пригодность объединенной конфигурации нужно подтверждать загрузкой в Конфигуратор 1С.

## Установка

Склонируйте репозиторий и установите пакет:

```bash
python -m pip install -e .
```

Для разработки и запуска тестов установите пакет с тестовыми зависимостями:

```bash
python -m pip install -e ".[dev]"
```

Проверьте, что команда доступна:

```bash
python -m cfmerge --help
cfmerge --help
```

## Запуск через графический интерфейс

После установки пакета можно открыть графический интерфейс:

```bash
cfmerge-gui
```

Если пакет не установлен как команда, запустите интерфейс напрямую из исходников:

```bash
python -m cfmerge.app
```

В окне нужно выбрать каталог `cf/`, каталог `cfu/` и каталог результата. Интерфейс запускает тот же механизм, что и команда `cfmerge merge`: строит `merged_cf/`, пишет JSON-отчет и текстовый отчет, показывает ход выполнения и может включать проверки XML/BSL и проверку через 1С.

## Готовый `exe` и самостоятельная сборка

В репозитории уже есть готовый файл:

```text
dist/1C_cf_cfu_merge.exe
```

Его можно запустить на Windows без ручного запуска Python-команд. Внутри это тот же графический интерфейс `cfmerge`.

Чтобы пересобрать `exe` самостоятельно, установите зависимости сборки и запустите скрипт:

```powershell
python -m pip install -e ".[build]"
powershell.exe -NoProfile -File ".\scripts\build-windows-exe.ps1"
```

Если зависимости уже установлены:

```powershell
powershell.exe -NoProfile -File ".\scripts\build-windows-exe.ps1" -SkipInstall
```

Скрипт пересоздает `dist/1C_cf_cfu_merge.exe`. Временные файлы PyInstaller складываются в `build/pyinstaller/` и не должны попадать в Git.

## Подготовка выгрузок 1С

Нужно подготовить два каталога:

```text
work/
  cf/
  cfu/
```

`cf/` - файловая выгрузка основной конфигурации.

`cfu/` - файловая выгрузка одного расширения.

Оба каталога должны быть выгружены из 1С в иерархическом формате файловой выгрузки.
Выгрузка конфигурации в файлы и слияние с расширением тестировались на платформе 8.3.27.1644.

## Быстрый пробный запуск

В репозитории есть небольшой синтетический пример:

```bash
python -m cfmerge merge \
  --cf examples/small/cf \
  --cfu examples/small/cfu \
  --out examples/small/merged_cf \
  --force \
  --report examples/small/merge-report.json \
  --write-human-report examples/small/merge-report.txt \
  --validate-xml \
  --validate-bsl
```

После выполнения появятся:

```text
examples/small/merged_cf
examples/small/merge-report.json
examples/small/merge-report.txt
```

Эти файлы являются результатами работы и не должны попадать в Git.

## Проверочный запуск без записи результата

Чтобы сначала посмотреть план действий, используйте `--dry-run`:

```bash
python -m cfmerge merge \
  --cf examples/small/cf \
  --cfu examples/small/cfu \
  --out examples/small/merged_cf \
  --dry-run \
  --report dry-run.json \
  --write-human-report dry-run.txt \
  --validate-xml \
  --validate-bsl
```

В этом режиме каталог `merged_cf/` не создается и не изменяется, но отчеты формируются.

## Пошаговый запуск на своей конфигурации

### 1. Подготовьте рабочий каталог

Пример структуры:

```text
work/
  cf/
  cfu/
```

### 2. Запустите пробный анализ

```bash
python -m cfmerge merge \
  --cf work/cf \
  --cfu work/cfu \
  --out work/merged_cf \
  --dry-run \
  --report work/dry-run.json \
  --write-human-report work/dry-run.txt \
  --validate-xml \
  --validate-bsl
```

Проверьте отчет. Если есть конфликты уровня `error`, результат строить нельзя, пока причина не разобрана.

### 3. Постройте объединенную конфигурацию

```bash
python -m cfmerge merge \
  --cf work/cf \
  --cfu work/cfu \
  --out work/merged_cf \
  --force \
  --report work/merge-report.json \
  --write-human-report work/merge-report.txt \
  --validate-xml \
  --validate-bsl
```

Параметр `--force` разрешает пересоздать каталог `work/merged_cf`, если он уже существует.

### 4. Проверьте отчет

Откройте:

```text
work/merge-report.txt
```

В первую очередь смотрите:

- `conflicts` - изменения, которые не были применены;
- `warnings` - места, требующие внимания;
- список добавленных объектов;
- список измененных модулей;
- разделы по формам: добавленные элементы, события, свойства, перемещения и пропущенные изменения.

### 5. Проверьте каталог скриптом `cf-validate.ps1`

```powershell
powershell.exe -NoProfile -File ".\tools\1c-cf-manage\scripts\cf-validate.ps1" `
  -ConfigPath "work\merged_cf"
```

Эта проверка не заменяет загрузку в 1С, но помогает быстро найти грубые ошибки структуры файловой выгрузки.

### 6. Загрузите результат в тестовую файловую базу

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-load-xml.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase" `
  -ConfigDir "work\merged_cf" `
  -Mode Full
```

Для серверной базы используйте `-InfoBaseServer` и `-InfoBaseRef`:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-load-xml.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>" `
  -ConfigDir "work\merged_cf" `
  -Mode Full
```

### 7. Примените конфигурацию к базе

Для файловой базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-update.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase"
```

Для серверной базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-update.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>"
```

### 8. Выгрузите готовую конфигурацию в `.cf`

Для файловой базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-dump-cf.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase" `
  -OutputFile "work\merged.cf"
```

Для серверной базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-dump-cf.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>" `
  -OutputFile "work\merged.cf"
```

Если загрузка, применение и выгрузка `.cf` прошли без ошибок, результат можно считать собранной обычной конфигурацией.

## Основные параметры команды `merge`

- `--cf` - путь к файловой выгрузке основной конфигурации.
- `--cfu` - путь к файловой выгрузке расширения.
- `--out` - путь к каталогу результата.
- `--force` - разрешить пересоздать каталог результата.
- `--dry-run` - выполнить анализ и построить отчеты без записи результата.
- `--report` - путь к JSON-отчету.
- `--write-human-report` - путь к текстовому отчету.
- `--validate-xml` - проверить, что XML результата разбирается и не содержит служебных признаков расширения.
- `--validate-bsl` - проверить, что в модулях результата не остались аннотации расширений и блоки вставки/удаления.
- `--validate-1c` - запустить локальные скрипты проверки через 1С, если указаны параметры платформы и базы.
- `--v8-path` - путь к каталогу `bin` платформы 1С.
- `--infobase-path` - путь к файловой тестовой базе.
- `--infobase-server` - сервер 1С для серверной тестовой базы.
- `--infobase-ref` - имя серверной тестовой базы.

## Как читать предупреждения

Предупреждение не всегда означает ошибку. Оно означает, что инструмент заметил место, которое нужно проверить внимательнее.

Пример:

```json
{
  "code": "FORM_PROPERTY_DELTA_NOT_APPLIED",
  "path": "Catalogs/ВидыДокументов/Forms/ФормаЭлемента/Ext/Form.xml"
}
```

Такое предупреждение означает: расширение изменило свойство формы, но инструмент не применил это изменение автоматически. В новых версиях часть таких изменений уже переносится, но отчет все равно нужно читать: если изменение не было применено, оно должно быть явно отражено в предупреждениях или конфликтах.
