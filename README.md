# cfmerge

`cfmerge` - инструмент для разработчиков 1С, который помогает превратить файловую выгрузку основной конфигурации и файловую выгрузку одного расширения в одну обычную конфигурацию 1С.

Обычная ситуация: есть каталог `cf/` с выгрузкой основной конфигурации и каталог `cfu/` с выгрузкой расширения. Расширение хранит изменения отдельным слоем: часть объектов заимствована из основной конфигурации, часть объектов добавлена самим расширением, модули содержат аннотации `&Перед`, `&После`, `&Вместо`, `&ИзменениеИКонтроль`, а формы могут содержать служебные данные расширения.

`cfmerge` строит каталог `merged_cf/`, который должен выглядеть как обычная файловая выгрузка конфигурации 1С, уже содержащая изменения расширения. Такой каталог можно загрузить в Конфигуратор через `/LoadConfigFromFiles`, применить к базе через `/UpdateDBCfg` и затем выгрузить в `.cf`.

## Что умеет инструмент

- Берет `cf/` как основу и не изменяет исходные каталоги.
- Добавляет собственные объекты расширения в результирующую конфигурацию.
- Не подменяет обычные XML-объекты заимствованными XML-обертками расширения.
- Переносит изменения модулей 1С:
  - `&Перед` и `&После` встраиваются в целевую процедуру;
  - `&Вместо` заменяет тело целевого метода;
  - `&Вместо` с `ПродолжитьВызов()` сохраняет вызов исходного метода через внутренний помощник;
  - `&ИзменениеИКонтроль` применяется только если исходный текст совпадает с ожидаемым.
- Переносит добавленные элементы, реквизиты, команды и события форм, когда это можно сделать безопасно.
- Удаляет из результата служебные признаки расширения: `callType`, `BaseForm`, аннотации расширений и блоки вставки/удаления.
- Регенерирует `ConfigDumpInfo.xml`.
- Объединяет `ОтчетПоКонфигурации.txt`.
- Формирует отчет в JSON и обычный текстовый отчет.
- Выполняет базовые проверки XML и модулей 1С.
- Может запустить локальные скрипты проверки через Конфигуратор 1С.

## Что понадобится

- Python 3.11 или новее.
- Платформа 1С:Предприятие 8.3 с `1cv8.exe`, если нужна финальная проверка загрузкой в базу.
- Файловая или серверная тестовая база 1С.
- Файловая выгрузка основной конфигурации.
- Файловая выгрузка расширения.

Проверки без запуска 1С можно выполнить только средствами Python. Но итоговую пригодность результата нужно подтверждать Конфигуратором 1С.

## Установка

Склонируйте репозиторий и установите пакет в режиме разработки:

```bash
python -m pip install -e .
```

Проверьте, что команда доступна:

```bash
python -m cfmerge --help
cfmerge --help
```

## Подготовка выгрузок 1С

Нужны два каталога:

```text
project/
  cf/
  cfu/
```

Где:

- `cf/` - файловая выгрузка основной конфигурации;
- `cfu/` - файловая выгрузка одного расширения.

Оба каталога должны быть выгружены в иерархическом формате файловой выгрузки 1С.

## Быстрый пробный запуск

В репозитории есть небольшой синтетический пример:

```bash
python -m cfmerge merge \
  --cf examples/small/cf \
  --cfu examples/small/cfu \
  --out examples/small/merged_cf \
  --force \
  --report examples/small/merge-report.json \
  --write-human-report examples/small/merge-report.txt \
  --validate-xml \
  --validate-bsl
```

После выполнения появится каталог:

```text
examples/small/merged_cf
```

И два отчета:

```text
examples/small/merge-report.json
examples/small/merge-report.txt
```

Эти файлы считаются результатами работы и не должны попадать в Git.

## Проверочный запуск без записи результата

Если нужно сначала посмотреть, что инструмент собирается сделать, используйте `--dry-run`:

```bash
python -m cfmerge merge \
  --cf examples/small/cf \
  --cfu examples/small/cfu \
  --out examples/small/merged_cf \
  --dry-run \
  --report dry-run.json \
  --write-human-report dry-run.txt \
  --validate-xml \
  --validate-bsl
```

В этом режиме `merged_cf/` не создается и не изменяется, но отчеты формируются.

## Пошаговый запуск на своей конфигурации

### 1. Подготовьте рабочий каталог

Пример структуры:

```text
work/
  cf/
  cfu/
```

### 2. Запустите пробный анализ

```bash
python -m cfmerge merge \
  --cf work/cf \
  --cfu work/cfu \
  --out work/merged_cf \
  --dry-run \
  --report work/dry-run.json \
  --write-human-report work/dry-run.txt \
  --validate-xml \
  --validate-bsl
```

Проверьте отчет. Если есть конфликты с уровнем `error`, результат строить нельзя, пока причина не разобрана.

### 3. Постройте объединенную конфигурацию

```bash
python -m cfmerge merge \
  --cf work/cf \
  --cfu work/cfu \
  --out work/merged_cf \
  --force \
  --report work/merge-report.json \
  --write-human-report work/merge-report.txt \
  --validate-xml \
  --validate-bsl
```

Параметр `--force` разрешает пересоздать каталог `work/merged_cf`, если он уже существует.

### 4. Проверьте отчет

Откройте текстовый отчет:

```text
work/merge-report.txt
```

На что смотреть в первую очередь:

- `conflicts` - ошибки, при которых изменение не применено;
- `warnings` - предупреждения, которые требуют внимания;
- список добавленных объектов;
- список измененных модулей;
- предупреждения по формам, если часть изменения формы не была перенесена автоматически.

### 5. Проверьте каталог выгрузки скриптом `cf-validate.ps1`

```powershell
powershell.exe -NoProfile -File ".\tools\1c-cf-manage\scripts\cf-validate.ps1" `
  -ConfigPath "work\merged_cf"
```

Эта проверка не заменяет загрузку в 1С, но помогает быстро найти грубые ошибки структуры выгрузки.

### 6. Загрузите результат в тестовую файловую базу

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-load-xml.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase" `
  -ConfigDir "work\merged_cf" `
  -Mode Full
```

Если база серверная, используйте `-InfoBaseServer` и `-InfoBaseRef`:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-load-xml.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>" `
  -ConfigDir "work\merged_cf" `
  -Mode Full
```

### 7. Примените конфигурацию к базе

Для файловой базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-update.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase"
```

Для серверной базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-update.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>"
```

### 8. Выгрузите готовую конфигурацию в `.cf`

Для файловой базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-dump-cf.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBasePath "C:\path\to\test-infobase" `
  -OutputFile "work\merged.cf"
```

Для серверной базы:

```powershell
powershell.exe -NoProfile -File ".\tools\1c-db-ops\scripts\db-dump-cf.ps1" `
  -V8Path "C:\Program Files\1cv8\8.3.xx.xxxx\bin" `
  -InfoBaseServer "<server-host>" `
  -InfoBaseRef "<infobase-ref>" `
  -OutputFile "work\merged.cf"
```

После этого файл `work/merged.cf` можно считать итоговой собранной конфигурацией, если все предыдущие команды завершились без ошибок.

## Основные параметры команды `merge`

- `--cf` - путь к файловой выгрузке основной конфигурации.
- `--cfu` - путь к файловой выгрузке расширения.
- `--out` - путь к каталогу результата.
- `--force` - разрешить пересоздать каталог результата.
- `--dry-run` - выполнить анализ и построить отчеты без записи результата.
- `--report` - путь к отчету JSON.
- `--write-human-report` - путь к текстовому отчету.
- `--validate-xml` - проверить, что XML результата разбирается и не содержит некоторых служебных признаков расширения.
- `--validate-bsl` - проверить, что в модулях результата не остались аннотации расширений и блоки вставки/удаления.
- `--validate-1c` - запустить локальные скрипты проверки через 1С, если указаны параметры базы и платформы.
- `--v8-path` - путь к каталогу `bin` платформы 1С.
- `--infobase-path` - путь к файловой тестовой базе.

## Как читать предупреждения

Предупреждение не всегда означает ошибку. Оно означает, что инструмент заметил место, которое может потребовать ручной проверки.

Пример:

```json
{
  "code": "FORM_PROPERTY_DELTA_NOT_APPLIED",
  "path": "Catalogs/ВидыДокументов/Forms/ФормаЭлемента/Ext/Form.xml"
}
```

Такое предупреждение означает: расширение изменило уже существующее свойство формы, например условное оформление. Инструмент это увидел, но не стал автоматически менять форму, потому что безопасно переносит в первую очередь добавленные элементы, команды, реквизиты и события. Такое место нужно проверить вручную в Конфигураторе.

## Что инструмент пока не умеет

- Не поддерживает несколько расширений одновременно и порядок их наложения.
- Не выполняет полное трехстороннее объединение для `&ИзменениеИКонтроль`, если исходный код основной конфигурации отличается от ожидаемого текста расширения.
- Не переносит автоматически все сложные изменения существующих свойств формы.
- Не гарантирует автоматический перенос удалений и перемещений элементов формы.
- Не является полноценным синтаксическим анализатором всего языка 1С. Анализ модулей ограничен тем, что нужно для переноса процедур, функций и аннотаций расширений.
- Не заменяет проверку Конфигуратором 1С. Успешные тесты Python и проверка XML не доказывают, что конфигурация применится к базе.
- Не решает ограничения конкретной СУБД. Например, файловая база может не применить конфигурацию из-за ограничений индексов, хотя серверная база примет ее.
- Не публикует и не содержит реальных производственных выгрузок. Примеры в репозитории предназначены только для демонстрации работы инструмента.
